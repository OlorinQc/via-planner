# Palantír 2.0 — Session 2 Report (write-path hardening)

Date: 2026-06-17. Run interactively (the original overnight run was cancelled with Fable 5;
nothing was lost). **No writes were made to Supabase.** All write-path testing was rehearsed
locally in PGlite against the real v1 backup.

---

## 1. Headline: keep SECURITY INVOKER (do not switch to DEFINER)

Decision resolved with live evidence:

- The work-chat Supabase connector executes as **`postgres`** (`current_user = postgres`,
  `rolbypassrls = true`), so it reads and writes regardless of RLS. It does not need DEFINER.
  (Session 1's SQL comment guessed `service_role`; it is actually `postgres`. Both bypass RLS,
  so the conclusion holds.)
- The React app executes as **`authenticated`**, and the `pal_auth_all` policy (USING/CHECK true)
  already permits all its writes.
- The only blocked caller is **`anon`**, which should stay blocked. Because `anon` currently
  holds EXECUTE on `pal_apply_update`, flipping that function to SECURITY DEFINER would turn it
  into an unauthenticated write endpoint at `/rest/v1/rpc/pal_apply_update`. That is a downgrade.

Invoker rights win, paired with the small hardening below.

## 2. Advisor findings (security)

- `function_search_path_mutable` (WARN) on all 12 `pal_` functions. **Fixed** (pin search_path).
- `rls_policy_always_true` (WARN) on all 9 `pal_` tables. **Accepted by design**: Palantír is
  single-user and the blueprint specified `authenticated/true`. Not changed.
- Out of scope, pre-existing, flagged only (not Palantír 2.0):
  `public.rls_auto_enable()` is a SECURITY DEFINER function executable by `anon`;
  `planner_state` (Work Planner) has the same permissive RLS; Auth leaked-password protection
  is off. Worth a look someday, but unrelated to this work.

## 3. Hardening test results (local PGlite)

Migrated the real v1 backup into a fresh PGlite (47 files, 217 task rows = 212 tasks + 5
milestones, 15 outputs, 12 people, 9 flags, 10 links), then ran the write path end to end.

- **27 / 27 behavior tests pass**: idempotency (same packageId = noop), people/file dedupe
  guards, title fallback + ambiguity skip, explicit-null due clearing, tasks complete/update/
  delete, referential safety on bad ids, flag default-kind + resolve-by-text, full
  filesToMerge (reassign + archive + moved counts), empty-package, export round-trip.
- **2 real bugs found** (robustness): unguarded `(ch->>'archived')::boolean` and
  `(ch->>'sortOrder')::float` casts throw on malformed input and, because the package is one
  transaction, **roll back every other change in the package**. A canary task created in the
  same package as a bad boolean was rolled back (confirmed). This is exactly the failure mode a
  chat-driven writer can trigger (e.g. `archived: "yes please"`, `sortOrder: "last"`).

Evidence trail: `session2-tests/` (harness.js, build_fixes.js, validate.js, session2-test-log.json).

## 4. The fix — `palantir-2.0-session2-fixes.sql` (drafted, NOT applied)

Generated by transforming the canonical Session 1 SQL, then **validated in PGlite: 13/13 pass.**
Contents, in apply order:

1. Two guarded helpers `pal_safe_bool(text)` and `pal_safe_float(text)` (catch + return NULL,
   matching the existing `pal_safe_date` / `pal_norm_flexdate` pattern).
2. A patched `pal_apply_update` that routes `archived` and `sortOrder` through those helpers:
   a malformed value is **ignored and warned** (`invalid archived value ignored: …`) instead of
   aborting. Valid values behave exactly as before (verified: archived true/false toggle,
   sortOrder 7.5 applies).
3. `search_path` pinned to `public, extensions, pg_temp` on all 14 functions (clears the 12
   advisor warnings; migrate and `gen_random_uuid` still resolve under the pinned path).
4. Least-privilege EXECUTE: revoke from `PUBLIC, anon` on `pal_apply_update`, `pal_snapshot`,
   `pal_export_state`; grant to `authenticated, service_role`. `pal_migrate_from_v1` locked to
   `service_role`. (`authenticated` is retained on snapshot/export because `pal_apply_update`
   is SECURITY INVOKER and calls them in-chain. PGlite runs as owner, so this ACL reasoning was
   verified by introspection, not by a behavioral RLS test, which is parked for the live run.)
